# 🧱 Setup nowego projektu (Git od zera)

---

## 📌 1. Tworzenie projektu

```bash
mkdir projekt

git init

📌 2. Ustawienie autora

👉 najlepiej global:
bash

git config --global user.name "Michal Baranski"

👉 opcjonalnie (dla konkretnego repo):
bash

git config user.email "inny@email.com"

bash

echo "hello" > README.md

bash

bash

git commit -m "pierwszy commit"

bash

git remote add origin git@github.com:user/repo.git

📌 7. Push
bash

git push -u origin main

📌 8. Praca na branchach
bash

git checkout -b feature/test

text

create → edit → add → commit → push → PR → merge

🔐 Git – SSH, klucze i autoryzacja
📌 1. Do czego jest SSH

👉 SSH służy do bezpiecznego łączenia się z repozytorium (GitHub/GitLab)

👉 zamiast: login + hasło ❌
👉 używasz: klucza SSH ✔
📌 2. Jak to działa

Masz 2 klucze:
🔑 prywatny (PRIVATE)

    zostaje u Ciebie

    NIGDY go nie wysyłasz

🔓 publiczny (PUBLIC)

    dodajesz do GitHub/GitLab

👉 działa to tak:

Ty (klucz prywatny) ↔ GitHub (klucz publiczny)

👉 jeśli pasują: 💥 masz dostęp
📌 3. Generowanie klucza
bash

ssh-keygen -t ed25519 -C "abmis@o2.pl"

👉 generuje:

    id_ed25519 (private)

    id_ed25519.pub (public)

📌 4. Dodanie do GitHub

👉 kopiujesz zawartość:
bash

cat ~/.ssh/id_ed25519.pub

👉 i wklejasz do: GitHub → Settings → SSH keys
📌 5. Test połączenia
bash

ssh -T git@github.com

👉 powinieneś zobaczyć: ✔ “You’ve successfully authenticated”
📌 6. Klonowanie przez SSH
bash

git clone git@github.com:user/repo.git

👉 NIE używasz: ❌ hasła, ❌ loginu
📌 7. Dlaczego SSH jest ważne

✔ bezpieczniejsze
✔ wygodniejsze
✔ automatyczne logowanie
✔ używane w firmach
📌 8. Błędy które już widziałeś

👉 No public key

oznacza:

    masz podpis

    ale system nie zna klucza

🧠 Git – konfiguracja, repo i workflow
📌 1. GLOBAL vs LOCAL (najważniejsze)
🌍 GLOBAL

    zapisany w: ~/.gitconfig

    działa na całym komputerze

    jest domyślny (fallback)

bash

git config --global user.name "Michal Baranski"
git config --global user.email "abmis@o2.pl"

📁 LOCAL

    zapisany w: repo/.git/config

    działa tylko w jednym repo

    nadpisuje global

bash

git config user.email "inny@email.com"

🧠 Jak Git wybiera email

    sprawdza LOCAL

    jak brak → bierze GLOBAL

    jak brak → ERROR

📌 2. COMMIT vs PUSH
✏️ COMMIT

    zapisuje zmiany lokalnie

    zapisuje autora (email + name)

bash

git commit -m "opis zmian"

👉 commit zawiera:

    autora

    datę

    wiadomość

🚀 PUSH

    wysyła commity na serwer

    NIE zmienia commitów

    NIE dodaje emaila

bash

git push origin main

🧠 Różnica

    commit = zapis zmian

    push = publikacja zmian

📌 3. Email w Git

    email zapisuje się w momencie commit

    NIE zmienia się później

    push nic nie zmienia

👉 commit zawsze ma autora
👉 bez autora → Git wyrzuca error
📌 4. Repo i remote
📁 Repo lokalne

    folder z .git

    przechowuje historię

🌐 Remote

    repo na serwerze (GitHub/GitLab)

bash

git remote -v

👉 może być wiele remote, ale zwykle jest:

    origin (główne repo)

📌 5. Realny workflow
text

clone → branch → commit → push → PR → review → merge

📌 6. Minimalny setup (praktyka)
bash

git config --global user.name "Michal Baranski"
git config --global user.email "abmis@o2.pl"

👉 i tyle wystarczy do pracy
📌 7. Kluczowe wnioski

💥 global = domyślne ustawienie na komputerze
💥 local = override dla repo

💥 commit zapisuje autora
💥 push tylko wysyła

💥 repo powinny być oddzielone (1 folder = 1 repo)
