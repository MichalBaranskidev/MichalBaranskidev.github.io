## <span id="setup-nowego-projektu">Setup nowego projektu</span>

### 📌 1. Tworzenie projektu

```bash
mkdir projekt
cd projekt
git init

📌 2. Ustawienie autora
<div class="tip-box"> ✅ <strong>Global (domyślny na całym komputerze)</strong> </div>
bash

git config --global user.name "Michal Baranski"
git config --global user.email "abmis@o2.pl"

<div class="note-box"> 📌 <strong>Opcjonalnie – tylko dla konkretnego repo (nadpisuje global)</strong> </div>
bash

git config user.email "inny@email.com"

📌 3. Tworzenie pliku
bash

echo "hello" > README.md

📌 4. Staging
bash

git add .

<div class="note-box"> 📌 <strong>Przygotowanie do commita</strong> – dodaje zmiany do "strefy przygotowania". </div>
📌 5. Commit
bash

git commit -m "pierwszy commit"

<div class="note-box"> 📌 Commit zapisuje:<br> ✔ autora (email + name)<br> ✔ datę<br> ✔ wiadomość </div>
📌 6. Podłączenie do remote
bash

git remote add origin git@github.com:user/repo.git

📌 7. Push
bash

git push -u origin main

📌 8. Praca na branchach
bash

git checkout -b feature/test

📌 9. Standardowy flow
text

create → edit → add → commit → push → PR → merge

📌 10. Najważniejsze zasady
Zasada	Opis
1 repo = 1 projekt	Każde repo osobny folder
1 commit = jedna zmiana	Małe, logiczne commity
używaj branchy	Feature branch do każdej zmiany
używaj PR do kontroli	Pull Request przed merge
📌 11. Największe błędy (do unikania)
<div class="warning-box"> ⚠️ <strong>Unikaj:</strong><br> ❌ mieszanie repo w jednym folderze<br> ❌ brak konfiguracji email<br> ❌ robienie wszystkiego na main<br> ❌ push bez sprawdzenia zmian </div>
<span id="ssh--klucze-i-autoryzacja">SSH – klucze i autoryzacja</span>
📌 1. Do czego jest SSH
<div class="tip-box"> ✅ SSH służy do bezpiecznego łączenia się z repozytorium (GitHub/GitLab) </div>

👉 zamiast: login + hasło ❌
👉 używasz: klucza SSH ✔
📌 2. Jak to działa

Masz 2 klucze:

🔑 prywatny (PRIVATE) – zostaje u Ciebie, NIGDY go nie wysyłasz
🔓 publiczny (PUBLIC) – dodajesz do GitHub/GitLab

👉 działa to tak:
Ty (klucz prywatny) ↔ GitHub (klucz publiczny)

👉 jeśli pasują: 💥 masz dostęp
📌 3. Generowanie klucza
bash

ssh-keygen -t ed25519 -C "abmis@o2.pl"

<div class="note-box"> 📌 Generuje:<br> ✔ <code>id_ed25519</code> (private)<br> ✔ <code>id_ed25519.pub</code> (public) </div>
📌 4. Dodanie do GitHub

👉 kopiujesz zawartość:
bash

cat ~/.ssh/id_ed25519.pub

👉 i wklejasz do: GitHub → Settings → SSH keys
📌 5. Test połączenia
bash

ssh -T git@github.com

<div class="tip-box"> ✅ powinieneś zobaczyć: "You've successfully authenticated" </div>
📌 6. Klonowanie przez SSH
bash

git clone git@github.com:user/repo.git

👉 NIE używasz: ❌ hasła, ❌ loginu
📌 7. Dlaczego SSH jest ważne

✔ bezpieczniejsze
✔ wygodniejsze
✔ automatyczne logowanie
✔ używane w firmach
📌 8. Błędy które już widziałeś
<div class="warning-box"> ⚠️ <code>No public key</code> oznacza:<br> - commit jest podpisany<br> - ale system nie zna klucza do weryfikacji<br> - <strong>NIE jest to błąd krytyczny</strong> </div>
<span id="git--konfiguracja-repo-i-workflow">Git – konfiguracja, repo i workflow</span>
📌 1. GLOBAL vs LOCAL
Typ	Lokalizacja	Zakres	Priorytet
🌍 GLOBAL	~/.gitconfig	cały komputer	niski (fallback)
📁 LOCAL	repo/.git/config	tylko jedno repo	wysoki (nadpisuje)
bash

# Global
git config --global user.name "Michal Baranski"
git config --global user.email "abmis@o2.pl"

# Local (w repo)
git config user.email "inny@email.com"

<div class="tip-box"> 🧠 <strong>Jak Git wybiera email:</strong><br> 1. sprawdza LOCAL<br> 2. jak brak → bierze GLOBAL<br> 3. jak brak → ERROR </div>
📌 2. COMMIT vs PUSH
Komenda	Co robi	Gdzie
git commit	zapisuje zmiany + autora	lokalnie
git push	wysyła commity na serwer	zdalnie
<div class="note-box"> 📌 <strong>Różnica:</strong><br> - commit = zapis zmian<br> - push = publikacja zmian<br> - push NIE zmienia commitów, NIE dodaje emaila </div>
📌 3. Email w Git

    email zapisuje się w momencie commit

    NIE zmienia się później

    push nic nie zmienia

<div class="warning-box"> ⚠️ Commit zawsze ma autora. Bez autora → Git wyrzuca error. </div>
📌 4. Repo i remote

📁 Repo lokalne – folder z .git, przechowuje historię
🌐 Remote – repo na serwerze (GitHub/GitLab)
bash

git remote -v

👉 może być wiele remote, ale zwykle jest: origin (główne repo)
📌 5. Realny workflow
text

clone → branch → commit → push → PR → review → merge

📌 6. Minimalny setup (praktyka)
bash

git config --global user.name "Michal Baranski"
git config --global user.email "abmis@o2.pl"

<div class="tip-box"> ✅ i tyle wystarczy do pracy – reszta to opcjonalne nadpisywanie. </div>
📌 7. Kluczowe wnioski
Wniosek	Wyjaśnienie
💥 global = domyślne ustawienie na komputerze	fallback, gdy nie ma local
💥 local = override dla repo	tylko tam, gdzie trzeba inaczej
💥 commit zapisuje autora	email + name w historii
💥 push tylko wysyła	nie zmienia commitów
💥 repo = osobny folder	1 folder = 1 repo
<span id="podsumowanie--tldr">Podsumowanie – TL;DR</span>
<div class="tip-box"> ✅ <strong>Zapamiętaj:</strong><br> ✔ ustaw global → żeby wszystko działało<br> ✔ używaj local tylko gdy potrzebujesz innego maila<br> ✔ commit ≠ push<br> ✔ email zapisuje się w commit<br> ✔ Git zawsze musi wiedzieć kim jesteś </div>
